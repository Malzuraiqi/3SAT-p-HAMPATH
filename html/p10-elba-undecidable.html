ELBA is Undecidable

A step-by-step proof using computation histories and the LBA BM,w construction.

CLO-2 R1 · 30 pts L7 6 Slides
Slide 1 · The Big Picture

The Undecidability of ELBA

Goal of this proof
We want to show that no algorithm can decide whether an LBA accepts any strings at all. The strategy: assume such an algorithm exists, then show it would let us solve a problem already proven impossible — a contradiction.
What is ELBA?

ELBA = { ⟨B⟩ : L(B) = ∅ }
"The set of all LBAs whose language is empty — they accept nothing."

What is an LBA?

A Linear Bounded Automaton is a Turing Machine that can only use the tape cells its input was written on — no extra space.

Figure · LBA input tape — the boundary is fixed

The Key Insight

We craft BM,w so that L(BM,w) ≠ ∅ if and only if M accepts w. That one equivalence makes the whole reduction work. The rest of this lecture explains how to build B.

ALBA vs ELBA — a surprising contrast

ALBA is decidable — "does B accept this specific input w?" is fine: B can only use |w| cells, so only finitely many configurations exist. Simulate and detect loops.

ELBA is undecidable — "does B accept any input?" means checking infinitely many possible strings. No space bound helps here.

Slide 2 · The Key Tool

Computation Histories

Core concept
An accepting computation history is a written record of every step a Turing Machine M went through to accept input w — from the very first step to the final accepting state. If someone hands you this record, you can check whether M really accepted w without ever running M yourself.

What is a Configuration?

At any moment during a TM's run, its full situation can be captured in a single string called a configuration. It packages three things: the tape contents, where the head is, and the current state.

Putting It All Together: The History String

A computation history just lists all the configurations in order, separated by #:

C₁ = q₀w
#

#

Three Rules a Valid History Must Follow

Rule 1 · Start right
The first configuration C₁ must be q₀w — the machine in its start state with head at the left, tape containing just w.
Rule 2 · Each step is legal
Every pair of consecutive configurations (Cᵢ, Cᵢ₊₁) must differ by exactly one application of M's transition function δ. No skipping steps, no illegal moves.
Rule 3 · End in accept
The last configuration Cₗ must contain qaccept. That's what it means for M to have accepted.
Why histories are useful

Each rule only requires looking at a small part of the string at a time — you never need to run M. Just read the record and check it. This is exactly what an LBA can do, and that's the whole trick.

Slide 3 · The Custom LBA B

Constructing BM,w

The construction
Given ⟨M, w⟩, we write down the description of a brand-new LBA called BM,w. We never run B during the reduction — we just design it. B has one job: check whether its input x is a valid accepting history for M on w.
The construction f(⟨M, w⟩) = ⟨BM,w⟩

BM,w is an LBA that, on input string x, does these four checks:

Check 1 · Structure
Is x in the form C₁ # C₂ # … # Cₗ? Scan for # separators dividing x into blocks.
Check 2 · Start
Does C₁ = q₀w? The correct start is fixed — B knows q₀ and w from when it was built.
Check 3 · Transitions
For every consecutive pair (Cᵢ, Cᵢ₊₁), does Cᵢ₊₁ follow from Cᵢ by one step of M's δ?
Check 4 · Accept state
Does the last block Cₗ contain qaccept?

All four pass → B accepts x. Any check fails → B rejects x.

The Bridge: What B's Language Tells Us About M

What M does on w L(BM,w) is …
M accepts w Not empty
M does not accept w Empty
Key Claim: L(BM,w) ≠ ∅  ⟺  M accepts w.
If M accepts, the sequence of configurations it went through is a valid history — that string lives in L(B). If M doesn't accept, no valid accepting history can exist, so nothing passes B's checks.
Slide 4 · Inside the Gears

How B Checks Transitions Without Extra Space

The crucial question
An LBA cannot use more tape than its input. How does B verify that Cᵢ₊₁ follows from Cᵢ correctly, given that both configurations are already written on its tape as part of x? The answer: B's head zig-zags back and forth between the two blocks, using small tape markers — no extra cells needed.
Zig-zag: verifying one (Cᵢ, Cᵢ₊₁) transition step-by-step Auto-plays · loops
Ready

B checks whether Cᵢ₊₁ follows from Cᵢ by a single δ-step. It does this by bouncing its head back and forth between the two configurations, using only the existing tape cells.

Why This Fits Within LBA Space

Everything is already there
The whole history string x is B's input. Both Cᵢ and Cᵢ₊₁ are already written on the tape. B doesn't need to generate anything new.
Markers use existing cells
B marks a symbol by overwriting it with a tagged version (e.g. a with ā). The marker lives in the same cell — no new tape is consumed.
Only a small window is checked
The transition δ only changes 1–3 characters around the head position. B's finite control remembers what it just read; it doesn't need scratch space.
Repeat for all pairs
Once a pair passes, B clears its marker and moves to the next # boundary. One pair at a time, all within the input tape.
Slide 5 · The Formal Proof

The Full Proof: ATM ≤m ELBA

The contradiction
Assume for contradiction that a decider R exists for ELBA. We construct machine S that uses R to decide ATM. Since ATM is undecidable, S cannot exist — so R cannot exist either.
Formal description of Machine S

S = "On input ⟨M, w⟩ where M is a TM and w is a string:"

  1. Construct the description ⟨BM,w⟩ as defined in Slide 3. (Just write the description — don't run B.)
  2. Run R on input ⟨BM,w⟩.
  3. If R says "L(B) = ∅" (empty) → reject.
  4. If R says "L(B) ≠ ∅" (not empty) → accept."

Why the Inversion Works

R says about ⟨B⟩ R's Output Meaning (from Slide 3 bridge) So S outputs
L(B) = ∅ R accepts M does not accept w Reject
L(B) ≠ ∅ R rejects M does accept w Accept
⚡ The Contradiction

S correctly decides ATM for every input ⟨M, w⟩.
But ATM is undecidable — no such machine can exist.
Our construction of S was logically sound, so the only error is…
the assumption that R exists.

∴ ELBA is undecidable. □
Slide 6 · Pitfalls, Q&A & Further Reading

Common Mistakes & Exam Defence

⚠️ Pitfalls Students Stumble On

Pitfall 1
f transforms — it doesn't execute
The reduction f only writes the description ⟨BM,w⟩. We never run B during the reduction. Running happens only when oracle R is invoked.
Pitfall 2
Don't lose the negation
The reduction targets ¬ELBA (not-empty). Always keep track: ⟨M, w⟩ ∈ ATM ⟺ ⟨B⟩ ∉ ELBA. That's why S inverts R's answer.
Pitfall 3
B verifies — it doesn't generate
Generating a history requires simulating M (needs unbounded tape). B only checks a history that's handed to it as input x. This separation is the entire technique.
Pitfall 4
"Linear bounded" means in the input size
BM,w's tape scales with |x|, the candidate history. Longer histories = longer inputs = more LBA tape. Still linear in the input. Always fits.

Q&A Defence Notes

  • Q1. Why CAN an LBA check accepting computation histories?
    All four checks are local: parsing # separators, matching the start config, validating each (Cᵢ, Cᵢ₊₁) δ-step using a 2–3 character window, and checking the accept state. Local checks fit in finite-state control plus a few tape sweeps — exactly the LBA budget.
  • Q2. Why is ALBA decidable but ELBA isn't?
    ALBA asks about one specific input w: space is bounded by |w|, so only finitely many configurations exist. Simulate and detect loops. ELBA asks whether B accepts any input — a question over infinitely many strings with no fixed length bound.
  • Q3. What's the bound on an LBA computation history size?
    An LBA on input of length n has at most |Q| · n · |Γ|n distinct configurations — exponential in n but finite for each fixed n. If any config repeats, it's a loop (how ALBA's decider catches non-halters).
  • Q4. Does this also prove ALLLBA undecidable?
    Yes. Complement BM,w (swap accept/reject). Then "language is everything" replaces "language is empty," with the negation flowing through. Many natural LBA properties follow this pattern — it's the LBA analog of Rice's theorem.
  • Q5. Why use BM,w instead of just simulating M?
    An LBA cannot simulate an arbitrary TM — that would require unbounded scratch space. The LBA only verifies a candidate history, which lives entirely within the input length. Verify vs. compute: this separation is the key.
